# PRD — 파수꾼 (가칭) · 에러 관제 SaaS

> 미니 Sentry + 미니 BetterStack + AI 장애 조사관.
> 어떤 Node.js 서비스든 SDK 한 줄로 붙이면, 에러를 모아 묶고, 능동적으로 감시하고, 장애가 나면 AI가 1차 조사까지 끝낸 리포트와 함께 알람을 울려주는 서비스.
> 작성일: 2026-07-10 · 작성: 이창호 + Claude

---

## 1. 문제 정의 (P)

- 운영 중인 실서비스 2개(진로나침반, chungaba.com)는 사용자가 제보해야 버그를 알 수 있다. 로그는 서버 안에만 쌓이고, 아무도 보고 있지 않다.
- 서버가 살아있고 HTTP 200이 떠도 장애인 경우가 있다. 업타임 핑 모니터링으로는 이런 장애를 절대 못 잡는다. (실무자 스레드에서 확인한 실제 사례)
- Discord 알람은 자는 사람을 깨우지 못한다. 심각한 장애는 전화가 와야 한다.
- 같은 장애를 두 번 겪어도 지난번에 어떻게 고쳤는지 기록이 없어 처음부터 다시 조사한다.

## 2. 목표 / 비목표

### 목표
1. 범용: 어떤 서비스든 프로젝트 등록 → API 키 발급 → SDK 장착으로 붙일 수 있다
2. 수동 수집: 에러/로그를 받아 같은 원인끼리 묶고(그룹핑) 새 에러·급증을 탐지한다
3. 능동 감시: 등록된 핵심 플로우를 주기적으로 실제 호출해 200이어도 깨진 장애를 잡는다
4. 알람: Discord → (무응답/심각) → 전화 에스컬레이션
5. AI: 유사 장애 검색(RAG) + 장애 조사 에이전트가 1차 조사 리포트를 알람에 첨부
6. 지식 축적: 해결 메모가 쌓여 장애 대응 런북이 자동으로 만들어진다
7. 첫 고객은 진로나침반과 chungaba.com (도그푸딩으로 실사용 증명)

### 비목표 (안 하는 것)
- 프론트엔드 에러 수집 (브라우저 SDK) — Node.js 백엔드 전용으로 시작
- 멀티테넌트 과금/결제
- 대규모 클러스터 (Kafka 브로커 1대, 단일 리전이면 충분)

## 3. 사용자

- 1인~소규모 팀으로 Node.js 서비스를 운영하는 개발자 (= 창호 본인이 페르소나)
- 화면: ① 에러 그룹 인박스 ② 장애 타임라인 ③ 합성 테스트 현황 ④ 프로젝트/알람 설정

## 4. 아키텍처

```
[고객 서비스 (jinro, chungaba)]
   └─ SDK: Winston 커스텀 transport (npm 배포)
        · 비동기 배치 전송, 실패 시 조용히 버림 (본 서비스에 절대 영향 금지)
        ▼
[인입 API]  (NestJS)  ── 받아서 Kafka에 넣기만 함 (폭주 대비 버퍼)
        ▼
[Kafka]  topic: events.raw
        ├─ 컨슈머 A: 그룹핑 (fingerprint) → MySQL 저장
        ├─ 컨슈머 B: 탐지 (신규 그룹 / 급증) → 알람 파이프라인
        └─ 컨슈머 C: 임베딩 → Qdrant (유사 장애 검색용)
        ▼
[알람 파이프라인]
   신규/급증 → AI 조사 에이전트 (수집→RAG 검색→가설→반증) → Discord 리포트 알람
   심각도 최상 + N분 무응답 → Solapi 보이스 API 전화
        ▲
[합성 테스트 워커]  등록된 시나리오를 5분 주기 실행, 실패 시 알람 파이프라인으로
[관제의 관제]  OpenTelemetry + Prometheus + Grafana/Loki (자기 자신 감시)
[배포]  Terraform → AWS ECS Fargate (인입 API / 컨슈머 / 합성 워커 분리) + ALB
```

## 5. 기술 스택과 선택 이유 (면접 답변 그대로)

| 기술 | 선택 이유 (왜) | 기존 경험과의 관계 |
|---|---|---|
| NestJS + TypeScript | 멘토 지침: 잘해야 하는 기본기 | 심화 |
| Kafka (KafkaJS) | 알람 컨슈머·분석 컨슈머·임베딩 컨슈머가 같은 이벤트를 각자 소비해야 함 → 작업 큐(BullMQ)가 아니라 이벤트 로그가 필연. 재소비 가능 | 신규 (BullMQ와 비교 설명 가능해짐) |
| MySQL + TypeORM | 국내 채용 공고 최다 조합을 경험 목적 (정직한 이유) | 신규 (Prisma+PG와 비교) |
| Winston | transport 구조가 SDK의 핵심 (커스텀 transport를 npm으로 배포) | 신규 (Pino와 비교 실측 글감) |
| Qdrant | 유사 장애 검색용 전용 벡터 DB | 신규 (pgvector와 비교) |
| OpenTelemetry + Prometheus + Grafana + Loki | 2026 관측 표준 (CNCF Graduated). 관제 서비스 자기 자신의 관측 | 신규 |
| ECS Fargate + ALB + Terraform | 멘토 지침(ECS) + EC2 단일 인스턴스의 순단·수동복구 한계 해결 + IaC | 신규 |
| Claude API (조사 에이전트) | 쇼핑 컨시어지의 모델 다운시프트·검증 패턴 재사용 | 재사용 (방법론 이식) |
| Solapi 보이스 API | 전화 알람 (통당 200원 → 심각도 게이트 설계 필요) | 신규 |

## 6. 기능 명세 (Phase별)

### Phase 0 — 세팅 (0.5주)
- [ ] 리포 생성, NestJS 스캐폴딩, docker-compose (Kafka, MySQL, Qdrant, Grafana 스택)
- [ ] 로그 표준 문서: 무엇을 남기고(레벨, traceId, 사용자 해시) 무엇을 절대 안 남기는지(PII, 토큰)

### Phase 1 — MVP: 수집과 그룹핑 (1주)
- [ ] 프로젝트 등록 + API 키 발급 (키는 해시 저장)
- [ ] 인입 API: POST /ingest (배치 수신, API 키 인증, 요청 검증) → Kafka 발행만
- [ ] 그룹핑 컨슈머: 스택트레이스 fingerprint 추출 → 에러 그룹 upsert + 발생 카운트
  - fingerprint 규칙: 에러 클래스 + 정규화된 스택 상위 N프레임 (경로의 행번호·해시 제거)
- [ ] 조회 API: 그룹 목록(발생수·최근발생·상태), 그룹 상세(이벤트 샘플)
- [ ] 신규 그룹 발생 시 Discord 웹훅 알람

### Phase 2 — SDK와 도그푸딩 (0.5주)
- [ ] Winston 커스텀 transport npm 패키지 (@pasukkun/winston-transport 가칭)
  - 비동기 배치(5초/50건), 실패 시 드롭, 전역 예외 훅 옵션
- [ ] 진로나침반 + chungaba.com에 실장착 (첫 실데이터)

### Phase 3 — 능동 감시 (1주)
- [ ] 급증 탐지 워치독: 1분 주기, 그룹별 최근 발생률 vs 기준선 비교
- [ ] 합성 테스트: 시나리오 등록(URL, 메서드, 기대 응답) → 5분 주기 실행 → 실패 시 알람
- [ ] 배포 스모크 테스트 가이드: 배포 파이프라인 끝에서 합성 테스트 즉시 1회 트리거하는 API

### Phase 4 — AI: RAG + 조사 에이전트 (1~1.5주)
- [ ] 해결 메모: 그룹 resolve 시 원인/조치 한 줄 기록 (지식베이스의 씨앗)
- [ ] 시드 데이터: 진로나침반·chungaba 과거 트러블슈팅 기록(Notion) 이관
- [ ] 유사 장애 검색 (Hybrid RAG): 에러 메시지 벡터(Qdrant) + 스택 키워드 결합
- [ ] 조사 에이전트: 신규/급증 트리거 시에만 실행 (비용 게이트)
  1. 수집: traceId 주변 로그, 최근 배포, 동시간대 다른 에러
  2. 검색: 유사 장애 + 해결 메모
  3. 가설: 원인 후보 + 심각도
  4. 반증: 별도 검증 에이전트가 가설을 공격
- [ ] A/B 측정: 단일 조사 vs 반증 포함, 리포트 유용성 직접 라벨링 → 결과대로 채택

### Phase 5 — 전화 에스컬레이션 (0.5주)
- [ ] 알람 정책: 심각도 룰 + Discord 무응답 N분 → Solapi 보이스 전화
- [ ] 비용 가드: 전화는 심각도 최상만, 일일 상한

### Phase 6 — 관측과 배포 (1주)
- [ ] 자기 관측: OTel 트레이스(인입→Kafka→컨슈머 전파), prom-client 메트릭(컨슈머 랙, 처리율), Winston→Loki, Grafana 대시보드
- [ ] Terraform: ECS Fargate 3개 서비스(api/consumer/synthetic) + ALB + ECR
- [ ] 무중단 검증: 배포 중 부하(autocannon) 걸고 5xx 0건 측정 (수치 확보)

### Phase 7 — 대시보드 UI (0.5~1주)
- [ ] Claude 디자인 스킬 스택 순서로: frontend-design → taste-skill → animate → playwright-mcp
- [ ] 디자인 시스템: 딥 네이비 B2B 팔레트 (수집한 Threads 프롬프트 활용)

### Phase 8 — 확장 (여유 시)
- [ ] 에러↔배포↔모듈 관계 지식그래프 (AX 다이어그램 방향)
- [ ] MCP 서버: Claude에서 오늘 장애 요약 조회

## 7. 성공 지표 (측정해서 이력서에 넣을 수치)

- 장애 인지: 사용자 제보 의존 → 알람까지 N분 (합성 테스트 주기 5분 이내)
- 배포 중 5xx 0건 (ECS 무중단 검증, 부하 테스트로 측정)
- 조사 리포트 유용성: A/B 라벨링 결과 (쇼핑 컨시어지 방법론 재사용)
- SDK 오버헤드: 장착 전후 p95 응답시간 차이 측정 (영향 없음 증명)
- npm 패키지 배포 (다운로드 수는 부차)

## 8. 리스크와 적대적 체크

| 리스크 | 대응 |
|---|---|
| 스코프 폭발 (알면서 진행) | Phase 순서 엄수. Phase 1~3까지가 제품의 뼈대. AI는 그 위에만 쌓는다 |
| SDK가 고객 서비스를 느리게 함 | fire-and-forget 원칙. 배치·비동기·실패 시 드롭. 측정으로 증명 |
| 알람 스팸 (양치기 소년) | 그룹핑이 심장인 이유. 같은 에러 1,000건 = 알람 1건. 급증 기준선 튜닝 과정을 기록 |
| LLM 비용 폭발 | 신규/급증에만 트리거, 모델 다운시프트, 프롬프트 캐싱 |
| RAG 콜드 스타트 | 해결 메모 UX 필수 + Notion 트러블슈팅 기록 시드 |
| Kafka 로컬/운영 비용 | 로컬 docker 단일 브로커. 운영은 ECS 컨테이너 1대 또는 관리형 비교 후 결정(비용표 작성) |
| 전화 비용 | 통당 200원 → 심각도 최상 + 일일 상한 |

## 9. 예상 면접 질문 (만들면서 답을 준비)

1. Sentry 무료 티어가 있는데 왜 만들었나 → 벤치마킹하고 내부 동작(그룹핑·탐지·수집 파이프라인)을 이해하려 만들었고, 내 운영 서비스 2개가 실사용 중
2. 왜 BullMQ가 아니라 Kafka인가 → 다중 컨슈머 독립 소비 + 재소비. 둘 다 써봤으므로 비교 가능
3. 수집 서버가 죽으면 고객 서비스는 → SDK fire-and-forget, 측정치로 증명
4. 200이 떠도 장애인 걸 어떻게 아나 → 합성 테스트
5. 알람 임계치는 → 기준선 대비 급증, 튜닝 기록
6. 멀티 에이전트가 정말 나은가 → A/B 측정 결과로 답변 (정직하게)
