# 카나리 로그 표준 (Phase 0 산출물)

카나리로 로그를 보내는 쪽(진로나침반, chungaba, 그리고 카나리 자신)이 지키는 규칙.

## 레벨 기준

| 레벨 | 언제 | 예 |
|---|---|---|
| error | 요청이 실패했고 사용자가 영향을 받았다 | 500 응답, 외부 API 최종 실패, 처리 불가 예외 |
| warn | 이상 신호지만 요청은 성공했다 | 재시도 후 성공, 응답 지연 임계 초과, 폐기된 입력 |
| info | 흐름 추적용 사실 기록 | 서버 기동, 결제 완료 같은 비즈니스 이벤트 |
| debug | 개발 중에만. 운영 전송 금지 | 변수 덤프 |

카나리로는 error와 warn만 보낸다. info 이하는 각 서비스의 로컬/Loki 로그로만 남긴다.

## 반드시 넣는 필드

- traceId: 요청 진입 시 발급해 그 요청의 모든 로그에 같은 값 (AsyncLocalStorage로 전파)
- name, message, stack: 에러 원본 그대로 (가공하지 않는다. 가공은 카나리 서버가 한다)
- context.path, context.method: 어느 API에서 났는지
- context.userHash: 사용자 구분이 필요하면 원본 ID가 아니라 해시로

## 절대 넣지 않는 것

- 비밀번호, 토큰, API 키, 쿠키 원문
- 이름, 전화번호, 이메일 등 개인정보 원문
- 상담 내용, 메시지 본문 등 사용자 생성 콘텐츠
- 요청 바디 전체 덤프 (필요한 필드만 골라 넣는다)

이유: 로그는 DB보다 접근 통제가 느슨하고 오래 남는다. 로그 유출이 곧 개인정보 유출이 되면 안 된다.

## SDK 동작 원칙 (fire-and-forget)

- 로그 전송은 본 서비스의 요청 처리를 절대 막지 않는다: 비동기 배치(5초 또는 50건), 전송 실패 시 조용히 버린다
- 카나리 서버가 죽어 있어도 고객 서비스는 아무 영향이 없어야 한다
